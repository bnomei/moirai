## Context

Close the public/evidence gap after T001–T008. Export `Bundle` from the root while keeping
`BundleWriter` in the advanced `world` namespace. Export `StageId` and add checked
`Schedule::stage_id(label) -> Option<StageId>` plus `Schedule::stage_label(&StageId) ->
Result<&str, ScheduleError>` so the handle is obtainable without exposing indices. Align the
prelude with system-authoring essentials, including `EntityScratch`, but do not turn it into every
root export.

Rename the crate-private raw-index stage-label helper and its `App` call site so it cannot collide
with the checked public `StageId` method; do not generalize the public method over raw indices.

Move test-only mutation/exhaustion controls off inherent `World`/`Schedule` methods into extension
traits exported only as `moirai::testkit::{WorldTestExt, ScheduleTestExt}`. `WorldTestExt` has
`set_change_tick_for_test(&mut self, ChangeTick)`, `set_world_tick_for_test(&mut self, u64)`, and
typed `set_event_sequence_for_test<E: Clone + 'static>(&mut self, next_sequence: u64, closed: bool)
-> Result<(), WorldError>`; the event hook resolves the registered typed channel and never accepts a
raw event index. `ScheduleTestExt` has
`stage_flush_mode_for_test(&self, label: &str) -> Option<FlushMode>`. Implement these traits for
`World` and `Schedule`, compile their internal module under `cfg(any(test, feature = "testkit"))`,
and publicly re-export them only with the `testkit` feature. Core builds expose none of these inherent
methods. Preserve `StepIndex::from_raw` as an existing testkit constructor.

Repair `scripts/generate_parity_ledger.py` so one canonical mapping regenerates the checked-in
ledger, detects duplicate/unknown test symbols and nonexistent paths, and supports a non-mutating
`--check` mode. Strengthen ledger tests to verify every cited path and test symbol. Replace false
event/query mappings with the real T003/T004/T007 regressions. Update README quickstart, phase
status, roadmap, rustdoc, changelog, public/prelude/compile-fail tests, and phase/parity documents so
all claims describe the landed API and behavior. Do not claim downstream cutover or performance.
