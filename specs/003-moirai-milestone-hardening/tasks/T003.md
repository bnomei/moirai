## Context

Close four related terminal-lifecycle holes. The runner must check mutation poison after every system
even if the system catches `ChangeTickExhausted` and returns success. Panic cleanup must be RAII-safe
without relying on the optional std catch layer, clear World/run-context fixed-step state, end the
World run, discard deferred commands, and retain the first generic panic fault when an unwind-capable
default-feature host catches the panic.

`update_with` and `render_with` observation callbacks are part of the App operation boundary. A
callback panic follows the same terminal-fault policy. For a failed Update observation, clear the
frame's Update events before the unwind continues; for Render, preserve the documented topology-only
behavior and finish diagnostics consistently. Do not swallow panics.

## DoD

- Caught component, resource, and query tick-exhaustion errors still terminally fault the App before
  another system or successful operation completion.
- Std and default-feature unwind-capable hosts leave the World idle, commands discarded, and the
  first panic fault retained after a caught system panic.
- FixedUpdate panic clears both public World and internal RunContext fixed-step state.
- `update_with`/`render_with` callback panics rethrow after terminal cleanup; Update does not leak
  prior-frame events into the next frame.
- Existing abort-target and no_std build posture remains valid without unsafe code.

## Validation

Run focused lifecycle/event/hostile tests, `cargo test --no-default-features`, all-feature tests,
strict Clippy, and Rust 1.75 no-default check. Add a default-feature unwind regression fixture if
ordinary integration tests enable std. A fresh Sol/high validator must review every return/unwind
path and first-fault retention.

## Escalate If

Retaining a generic panic fault on unwind cannot be made safe without catch_unwind, unsafe code, or a
public callback contract change.
